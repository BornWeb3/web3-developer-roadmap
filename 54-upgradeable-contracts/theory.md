# День 54: upgradeable contracts и proxy

## в чем проблема обычных контрактов

После деплоя контракт нельзя изменить  

код уже находится в блокчейне  

если нашёлся баг  
или нужна новая функция  

просто переписать контракт нельзя  

## решение

upgradeable contracts  

Это система  
где логику можно обновлять  

## как это работает

используется proxy контракт  

пользователи взаимодействуют именно с ним  

а proxy перенаправляет вызовы  
в другой контракт с логикой  

## что хранит proxy

storage  
балансы  
состояние  

## что хранит implementation

код и функции  

## что происходит при обновлении

меняется implementation контракт  

но proxy остаётся тем же  

адрес не меняется  
данные тоже  

## почему это удобно

можно исправлять баги  
добавлять функции  
обновлять систему  

## как работает delegatecall

proxy использует delegatecall  

код выполняется из implementation  

но storage используется proxy  

## важный момент

storage layout должен совпадать  

иначе можно сломать данные  

## простой пример

в первой версии  

есть balance  

во второй версии  

порядок переменных изменили  

данные начинают читаться неправильно  

## риски

ошибки обновления  
сломанный storage  
небезопасный доступ к upgrade  

## кто может обновлять

обычно  

admin  
multisig  
governance  

## почему это опасно

если upgrade доступен атакующему  

он может заменить логику  
и украсть средства  

## пример из жизни

это как двигатель машины  

кузов остаётся тот же  

но двигатель можно заменить  

## главная мысль

proxy позволяет обновлять контракт  

данные остаются  
логика меняется  

это мощный инструмент  

но ошибки в upgrade системе  
могут быть очень опасными
