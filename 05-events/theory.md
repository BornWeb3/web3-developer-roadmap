# День 5: events и логи, как фронтенд видит контракт

## почему фронтенд вообще что то знает

Смарт контракт не может сам отправить данные на фронтенд  

Он не умеет пушить информацию наружу  

Фронтенд не сидит внутри блокчейна  
он только читает данные  

Возникает вопрос  
как тогда интерфейс узнаёт что произошло  

## что такое events

Event это запись о том что произошло внутри контракта  

Это не состояние  
это просто лог  

Пример  

пользователь отправил токены  
контракт может записать событие Transfer  

это как запись в журнале  

## как это работает

Когда транзакция выполняется  

контракт может сгенерировать event  

этот event записывается в логи транзакции  

и сохраняется в блокчейне  

Важно  

это не storage  
это отдельный слой данных  

## где хранятся events

Когда транзакция выполняется  
создаётся receipt  

в нём есть  

статус  
потраченный газ  
логи  

вот в этих логах и лежат events  

## из чего состоит event

Event состоит из двух частей  

topics  
data  

topics используются для поиска  

data содержит сами данные  

Пример  

event Transfer  

topics помогут быстро найти все Transfer  
data будет хранить адреса и сумму  

## почему это важно для фронтенда

Фронтенд слушает events  

он подписывается на нужные события  
и реагирует на них  

Пример  

ты отправил токены  

контракт сгенерировал event  
фронтенд увидел его  
и обновил баланс на экране  

## простой пример из жизни

Представь что контракт это сервер  

а events это лог файл  

фронтенд читает лог  
и понимает что произошло  

## почему не использовать только storage

Можно читать данные напрямую из storage  

но это неудобно  

events дают  

быстрый поиск  
удобную структуру  
историю действий  

Пример  

найти все переводы токенов  
через storage сложно  
через events легко  

## главная мысль

Events это журнал действий контракта  

они не хранят состояние  
а записывают что произошло  

фронтенд не видит контракт напрямую  
он читает events и понимает что происходит
