# День 15: events как data layer

## почему нужен отдельный слой данных

Storage хранит текущее состояние  

но он не даёт удобной истории  

Если тебе нужно понять что происходило раньше  
storage не поможет  

Пример  

ты видишь баланс сейчас  
но не видишь все переводы  

## что делают events

Events это журнал действий контракта  

они записывают  

что произошло  
кто вызвал  
какие данные были  

Это не состояние  
это история  

## почему это называют data layer

Потому что events используются как отдельный слой данных  

Фронтенд  
аналитика  
индексаторы  

все читают именно events  

а не storage напрямую  

## как это работает

Контракт выполняет действие  

и записывает event  

этот event попадает в логи транзакции  

и сохраняется в блокчейне  

## пример

пользователь отправил токены  

контракт изменил баланс  
и записал event Transfer  

фронтенд увидел event  
и обновил интерфейс  

## почему это удобно

events легко искать  

у них есть topics  
по которым можно фильтровать  

Пример  

найти все переводы конкретного пользователя  

через storage сложно  
через events быстро  

## как используется на практике

фронтенд подписывается на события  

или запрашивает историю  

и строит интерфейс  

Пример  

история транзакций  
баланс  
активность  

всё это часто строится через events  

## важный момент

events нельзя использовать как источник истины  

они не влияют на состояние  

это только запись  

если что то важно для логики  
это должно быть в storage  

## пример из жизни

storage это текущее состояние счёта  

events это выписка операций  

## главная мысль

events это журнал действий  

они не хранят состояние  
но дают удобный доступ к данным  

это отдельный слой  
через который фронтенд понимает что происходит
