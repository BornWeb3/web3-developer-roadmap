# День 13: proxy и upgradeable контракты

## в чём проблема обычных контрактов

Когда ты деплоишь контракт  
его нельзя изменить  

Код навсегда остаётся таким как есть  

Если ты нашёл баг  
или хочешь добавить новую функцию  
придётся деплоить новый контракт  

И это проблема  

потеряются адреса  
пользователи  
данные  

## решение

Upgradeable контракты  

Идея в том  
чтобы разделить  

данные  
и логику  

## что такое proxy

Proxy это контракт  
который хранит данные  

и просто перенаправляет вызовы  

Он не содержит основной логики  

## как это работает

Есть два контракта  

proxy контракт  
implementation контракт  

Пользователь взаимодействует только с proxy  

proxy принимает вызов  
и через delegatecall передаёт его в implementation  

Код выполняется из implementation  
но данные записываются в proxy  

## почему это важно

Можно менять implementation  
не трогая proxy  

Адрес остаётся тот же  
данные остаются  
меняется только логика  

## простой пример

есть контракт с токеном  

ты нашёл баг  
обновил implementation  

proxy теперь использует новый код  

но  

балансы пользователей остались  

## что происходит внутри

proxy хранит  

адрес implementation  

при вызове  

он берёт calldata  
делает delegatecall  
выполняет код  

## важный момент

storage должен совпадать  

если структура данных изменится неправильно  
можно сломать контракт  

## риски

неправильный upgrade может сломать всё  

если доступ к обновлению открыт  
можно заменить логику на вредоносную  

## пример из жизни

proxy это как оболочка приложения  

implementation это логика внутри  

ты обновляешь приложение  
но аккаунт и данные остаются  

## главная мысль

proxy разделяет данные и логику  

delegatecall позволяет выполнять код в storage proxy  

upgrade меняет логику без смены адреса  

это основа для гибких и обновляемых протоколов
