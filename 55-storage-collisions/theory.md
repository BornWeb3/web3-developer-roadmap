# День 55: storage collisions

## почему это важно

В upgradeable контрактах  

storage хранится отдельно от логики  

если storage организован неправильно  

данные могут сломаться  

## что такое storage collision

Это конфликт storage слотов  

когда разные переменные  

начинают использовать одно и то же место в памяти  

## как это происходит

proxy хранит данные  

implementation использует эти данные  

если в новой версии изменить порядок переменных  

storage layout ломается  

## простой пример

в первой версии  

uint balance  
address owner  

во второй версии  

address owner  
uint balance  

контракт начнёт читать данные неправильно  

## что происходит дальше

balance может стать owner  

owner может стать мусором  

контракт ломается  

## почему это опасно

storage это основа состояния контракта  

ошибка может уничтожить  

балансы  
права доступа  
настройки  

## где это особенно критично

proxy системы  
delegatecall  
upgradeable контракты  

## как защищаются

никогда не менять порядок storage  

не удалять переменные  

добавлять новые только в конец  

## что такое storage slot

каждая переменная хранится  

в определённом месте памяти  

slot 0  
slot 1  
slot 2  

## важный момент

delegatecall использует storage proxy  

а не implementation  

поэтому layout должен совпадать идеально  

## пример из жизни

представь склад  

где коробки подписаны  

если поменять порядок  

люди начнут брать не те вещи  

## главная мысль

storage collision это конфликт данных  

из за неправильного storage layout  

в proxy системах  

ошибка может полностью сломать контракт  

поэтому storage структура  
это критически важная часть upgradeable архитектуры
